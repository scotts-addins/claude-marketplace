# Chart of accounts intake

Every report this skill builds lists individual account codes from the user's chart of accounts. Don't guess code ranges. Don't assume conventions. The chart is the source of truth.

## How to ask for it

At the start of every workflow that produces a financial report, ask the user with this exact phrasing:

> Please supply an Excel workbook that contains your organization's chart of accounts generated by using the =SCOTT.CHART custom function. Be sure the workbook is saved in a connected / authorized state to your accounting system.

Do not paraphrase. The "saved in a connected / authorized state" qualifier matters: when `SCOTT.CHART` evaluates against a live, authenticated session, the result spills into the cells and the cached values are written into the saved file. If the user saved it while disconnected, the cells will contain `#VALUE` or stale errors and the chart isn't usable.

If the user says they don't have one ready, walk them through it briefly:

1. Open Excel with Scott's Add-ins active and signed in.
2. In a blank workbook, type `=SCOTT.CHART(112353)` (or their org id) into A1.
3. Wait for it to spill — it'll fill columns A through F or so with every account.
4. Save the workbook and share it back.

## What the workbook contains

The exact column layout depends on what `SCOTT.CHART` returns — confirm by reading the actual file the user supplies. A typical Xero export through `SCOTT.CHART` includes columns like:

- Account code
- Account name / description
- Account type (e.g., `BANK`, `CURRENT`, `FIXED`, `CURRLIAB`, `EQUITY`, `REVENUE`, `EXPENSE`, `OVERHEADS`, etc.)
- Tax type
- Reporting code

Don't assume column names — open the file with `openpyxl` (read-only mode) and read row 1 to find them. If the file has a header row and a spilled-array body below, the column headers are usually in row 1.

## Mapping account types to report sections

Once you've read the chart, map each account's type to the correct section of the report you're building. The exact type strings depend on the accounting system; for Xero through `SCOTT.CHART` the common mappings are:

| Type string in chart       | Goes on              | Section                 |
|----------------------------|----------------------|-------------------------|
| BANK                       | Balance Sheet        | Cash and Cash Equivalents |
| CURRENT (or CURRASSET)     | Balance Sheet        | Current Assets          |
| INVENTORY                  | Balance Sheet        | Current Assets          |
| PREPAYMENT                 | Balance Sheet        | Current Assets          |
| FIXED                      | Balance Sheet        | Fixed Assets            |
| NONCURRENT                 | Balance Sheet        | Long Term Assets        |
| CURRLIAB                   | Balance Sheet        | Current Liabilities     |
| LIABILITY (or NONCURRLIAB) | Balance Sheet        | Long Term Liabilities   |
| TERMLIAB                   | Balance Sheet        | Long Term Liabilities   |
| EQUITY                     | Balance Sheet        | Equity                  |
| REVENUE                    | P&L                  | Revenue                 |
| SALES                      | P&L                  | Revenue                 |
| DIRECTCOSTS                | P&L                  | Direct Costs            |
| EXPENSE                    | P&L                  | Operating Expenses      |
| DEPRECIATN                 | P&L                  | Operating Expenses      |
| OVERHEADS                  | P&L                  | Operating Expenses      |
| OTHERINCOME                | P&L                  | Other Income (placed after Operating Income) |

Some types may not appear in any given chart, and some charts have additional types not listed above. If you see a type string not in this table, ask the user where it should go rather than guessing.

## Reading the chart in Python

Use `openpyxl` for reading (xlsxwriter is write-only). The chart workbook is small, so loading it whole is fine:

```python
import openpyxl
wb = openpyxl.load_workbook('chart.xlsx', read_only=True, data_only=True)
ws = wb.active

# Find header row — usually row 1
headers = [c.value for c in ws[1]]
# Build a {column_name: index} map
col = {h: i for i, h in enumerate(headers) if h}

# Iterate body rows
accounts = []
for row in ws.iter_rows(min_row=2, values_only=True):
    if row[col['Code']] is None:
        continue
    accounts.append({
        'code': str(row[col['Code']]),
        'name': row[col['Name']],
        'type': row[col['Type']],
    })
```

`data_only=True` is essential — without it, openpyxl returns formula text (`=SCOTT.CHART(112353)`) instead of the cached spilled values. With `data_only=True` it returns whatever was last cached when Excel saved the file (which is why the "saved in a connected / authorized state" requirement exists).

## Building the section list

Once you have a list of accounts grouped by section, generate the report by iterating through each section and writing one `SCOTT.GL` (or `SCOTT.BUDGET`, etc.) row per account, exactly as `build_scott_pnl.py` does — but with the codes coming from the chart instead of being hardcoded. Section totals stay as ordinary `=SUM` over the rows in that section.

Order accounts within each section by code (numeric sort if the codes parse as integers, otherwise lexicographic). Don't try to match a specific report ordering from the user's accounting system unless they ask for it — the chart's natural order is the safe default.

## When the chart is missing or unusable

If the user can't supply a chart (e.g., they're offline, the auth flow is broken, the file they sent is empty or full of errors), tell them clearly that the report can't be properly built without it. As a last-resort fallback, you can build a range-based version using `SCOTT.RANGE` against conservative ranges, but you must:

1. Tell the user up front that this is a fallback approximation, not a real report.
2. List the ranges you used in the handoff message.
3. Note that any account outside those ranges will be missing from the totals.

Don't silently fall back. The user needs to know they're getting an approximation.
